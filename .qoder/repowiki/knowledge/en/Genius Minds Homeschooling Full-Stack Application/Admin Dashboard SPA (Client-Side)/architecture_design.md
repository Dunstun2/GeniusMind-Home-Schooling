Static client-side SPA served from `/admin/` with three entry points: `login.html` (standalone auth form posting to `/api/admin/login`), `index.html` (main dashboard shell), and shared `script.js` + `styles.css`. The dashboard is a single HTML page containing one `<section id="tab-<feature>">` per feature (dashboard, bookings, emails, analytics, users, banners, settings, pages, team, courses, faqs, blog, social); navigation is driven by `data-tab` attributes on sidebar links that toggle `.active` classes and call a `loadXxx()` fetcher per tab. All data operations go through REST endpoints under `/api/admin/*` (`/api/admin/stats`, `/api/admin/bookings`, `/api/admin/emails`, `/api/admin/analytics`, `/api/admin/users`, `/api/admin/whatsapp/status`, etc.). Authentication is enforced via a `/api/admin/check-session` probe on load; 401 responses redirect to `login.html`. Charts are rendered in-place on `<canvas>` elements using Chart.js (line/doughnut/polarArea/bar). WhatsApp integration is polled every 2 s during QR/auth states and every 20 s otherwise. A collapsible diagnostic panel exposes raw sessions/events tables fetched from `/api/admin/analytics`. There is no build step — files are plain HTML/CSS/JS served directly by the backend's static file handler.
